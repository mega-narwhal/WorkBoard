## 5. Study C — Live carding (steady-state cost)

### (1) Memory-write per session
| | model calls | input tok | over 100 sessions |
|---|--:|--:|--:|
| WorkBoard (inline carding) | 0 | 0 | 0 |
| claude-mem (SessionEnd compress) | 1 | 5,462 | 546,200 |

WorkBoard's carding writes text the model already produced — no extra call. claude-mem compresses every session (real run §8 confirmed: **main-tier** subscription model, not a cheap tier).

### (2) Context-injection per session
- WorkBoard: 97 SessionStart + 306/turn nudge = 15,397/50-turn (trimmable to 2,097); board never auto-loads → **constant in memory size**.
- claude-mem: ~thousands (grows with stored memory) at SessionStart, **grows** with stored memory.

### (3) The nudge tradeoff (breakeven)
WorkBoard's per-turn nudge is its one heavier surface. Breakeven vs claude-mem's per-session compression: **~18 turns** (full nudge) / **~137 turns** (trimmed). Caveat: nudge tokens are interactive; claude-mem's compression is a separate (main-tier) call. The clean win is **0 model calls** to persist.

### Per-recall (from Study B)
WorkBoard 2,399 vs claude-mem 3,237 (25.9% lighter) vs mem0 6,719 (64.3% lighter).

## 8. Where claude-mem wins (honest tradeoffs)

- **Vague / off-board recall.** Facts never carded (e.g. ['P06']) aren't on the board; claude-mem's vector store can surface them.
- **Tight single-fact pinpoints.** A compressed observation can be smaller than a content-rich card; claude-mem wins a few pinpoint queries (§4).
- **Cross-project + zero-structure capture.** WorkBoard is project-scoped and needs carding discipline; claude-mem captures everything automatically.

They are complements: claude-mem is conversational memory; WorkBoard is the project ledger of work outcomes. WorkBoard's edge is structured, deterministic recall + near-zero build & write cost.
